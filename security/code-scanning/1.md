## What CodeQL reported

`clearCookie` is the shared deletion helper: every cookie the auth package expires goes through it,
including the sensitive ones — the chunked session cookie `patchy-auth` (via
[`clearChunked`](https://github.com/bitwise-media-group/patchy/blob/b377012bb2c0038ad49f1ca7c66ac4dce7fba9dc/internal/web/auth/cookies.go#L98-L103))
and the OAuth CSRF state cookie `patchy-oauth2-state` (via
[`handleCallback`](https://github.com/bitwise-media-group/patchy/blob/b377012bb2c0038ad49f1ca7c66ac4dce7fba9dc/internal/web/auth/oidc.go#L170)).
The expiring replacement it writes sets no `HttpOnly` attribute:

```go
// clearCookie expires one cookie.
func clearCookie(w http.ResponseWriter, name string) {
    http.SetCookie(w, &http.Cookie{Name: name, Value: "", Path: "/", MaxAge: -1})
}
```

## Remediation

True positive, with bounded impact: the flagged `Set-Cookie` is a deletion (`MaxAge: -1`, empty
value), so there is no session material for a script to read from it. It is still the wrong posture —
the replacement cookie a clear writes should be at least as locked down as what it removes, and the
live writes already are
([`writeChunked`](https://github.com/bitwise-media-group/patchy/blob/b377012bb2c0038ad49f1ca7c66ac4dce7fba9dc/internal/web/auth/cookies.go#L72-L80)
sets `HttpOnly`, `Secure`, and `SameSite=Lax` on every session chunk).

The fix gives `clearCookie` the same attributes: `HttpOnly: true`, `Secure` threaded from the
existing `cfg.Insecure` plain-HTTP dev flag, and `SameSite=Lax`. Browser deletion matches on name and
path only, so the hardened attributes never prevent the clear.
`internal/web/auth/cookies_test.go` gained `TestClearCookieAttributes` pinning the new attributes.

## Resolution

same commit fixes both.
