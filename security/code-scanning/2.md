## What CodeQL reported

`setJSONCookie` writes the status page's SPA-visible cookies — `patchy-auth-provider`, `patchy-auth-error`, and
`patchy-auth-logout` — without the `HttpOnly` attribute. The rule matched because the cookie names contain `auth`, which
its sensitive-name heuristic treats as session material:

```go
// setJSONCookie writes an SPA-visible base64url JSON cookie. Not HttpOnly by
// design — the SPA reads it; it must never carry a secret.
func setJSONCookie(w http.ResponseWriter, name string, v any, maxAge time.Duration) error {
    raw, err := json.Marshal(v)
    if err != nil {
        return fmt.Errorf("cookie %s: %w", name, err)
    }
    http.SetCookie(w, &http.Cookie{
        Name:     name,
        Value:    base64.RawURLEncoding.EncodeToString(raw),
        Path:     "/",
        MaxAge:   int(maxAge.Seconds()),
        SameSite: http.SameSiteLaxMode,
    })
    return nil
}
```

## Why this is a false positive

1. **These cookies carry no secrets, by documented design.** The
   [cookie-name block](https://github.com/bitwise-media-group/patchy/blob/b377012bb2c0038ad49f1ca7c66ac4dce7fba9dc/internal/web/auth/cookies.go#L16-L18)
   and the
   [`setJSONCookie` doc comment](https://github.com/bitwise-media-group/patchy/blob/b377012bb2c0038ad49f1ca7c66ac4dce7fba9dc/internal/web/auth/cookies.go#L110-L111)
   both state the contract: SPA-visible, never a secret. The three payloads are a sign-in-surface descriptor
   (`{"provider","authenticated","autoLogin"}`, written by
   [`setProviderCookie`](https://github.com/bitwise-media-group/patchy/blob/b377012bb2c0038ad49f1ca7c66ac4dce7fba9dc/internal/web/auth/oidc.go#L303-L312)),
   a one-shot human-readable sign-in failure message (written by
   [`fail`](https://github.com/bitwise-media-group/patchy/blob/b377012bb2c0038ad49f1ca7c66ac4dce7fba9dc/internal/web/auth/oidc.go#L314-L320)),
   and a boolean logout marker (written by
   [`handleLogout`](https://github.com/bitwise-media-group/patchy/blob/b377012bb2c0038ad49f1ca7c66ac4dce7fba9dc/internal/web/auth/oidc.go#L229-L235)).
   Stealing any of them via XSS yields nothing a same-origin script could not already observe.
2. **`HttpOnly` would break the feature.** The SPA reads these cookies through `document.cookie` to render the sign-in
   surface and consumes the error/logout markers by deleting them client-side — see
   [`readCookie`](https://github.com/bitwise-media-group/patchy/blob/702aedd59a401a35fe136ee94356119a17d1c9a8/internal/web/ui/src/auth.ts#L20-L25)
   and
   [`deleteCookie`](https://github.com/bitwise-media-group/patchy/blob/702aedd59a401a35fe136ee94356119a17d1c9a8/internal/web/ui/src/auth.ts#L38-L40)
   (permalinked at `702aedd`, the later branch commit that adds the SPA source; it postdates the analyzed merge commit).
   An `HttpOnly` cookie is invisible to `document.cookie`, so the UI could never see it.
3. **The actually sensitive cookies are `HttpOnly`.** The session chunks
   ([`writeChunked`](https://github.com/bitwise-media-group/patchy/blob/b377012bb2c0038ad49f1ca7c66ac4dce7fba9dc/internal/web/auth/cookies.go#L72-L80))
   and the OAuth CSRF state cookie
   ([`handleAuthorize`](https://github.com/bitwise-media-group/patchy/blob/b377012bb2c0038ad49f1ca7c66ac4dce7fba9dc/internal/web/auth/oidc.go#L146-L154))
   both set `HttpOnly: true` (and `Secure`). The rule's premise — that a name containing `auth` implies session material
   — fails here; this is a name-heuristic match, not a traced flow of sensitive data.

The sibling `Secure` finding on the same call ([4](./4.md)) is a true positive and is being remediated; only the
`HttpOnly` half is by design.

## Resolution

Recommend dismissing in the code-scanning UI as **won't fix**, with a comment noting the cookies are SPA-visible by
design and carry no secrets while the session and OAuth state cookies remain `HttpOnly`. Outcome: **False positive**.
