## What CodeQL reported

The same `clearCookie` call flagged by finding [1](./1.md) also omits the `Secure` attribute on the expiring replacement
cookie it writes:

```go
// clearCookie expires one cookie.
func clearCookie(w http.ResponseWriter, name string) {
    http.SetCookie(w, &http.Cookie{Name: name, Value: "", Path: "/", MaxAge: -1})
}
```

## Remediation

True positive, same shape as finding [1](./1.md): the deletion cookie carries no value, so nothing confidential can leak
over plaintext HTTP, but the clear should mirror the attributes of the live writes —
[`writeChunked`](https://github.com/bitwise-media-group/patchy/blob/b377012bb2c0038ad49f1ca7c66ac4dce7fba9dc/internal/web/auth/cookies.go#L72-L80)
already derives `Secure` from the deployment's `cfg.Insecure` flag
([config](https://github.com/bitwise-media-group/patchy/blob/b377012bb2c0038ad49f1ca7c66ac4dce7fba9dc/internal/web/auth/config.go#L61-L63)).

The fix threads that same flag into `clearCookie` (and `clearChunked`), so production clears carry `Secure` while the
plain-HTTP local-dev posture keeps working — a hardcoded `Secure: true` would be rejected by browsers on non-TLS dev
origins and the clear would silently stop working there.

## Resolution

Remediated in commit
[`fb1bc98`](https://github.com/bitwise-media-group/patchy/commit/fb1bc98e8f117e6a939489f0f89b2be7f863d8c9) on the PR #47
branch (`feat/progress-ui`); the alert closes when CodeQL re-analyzes with the fix merged. The change is shared with
finding [1](./1.md) (`HttpOnly` on the same line).
