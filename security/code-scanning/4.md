## What CodeQL reported

`setJSONCookie` — the writer of the SPA-visible `patchy-auth-provider`, `patchy-auth-error`, and `patchy-auth-logout`
cookies — omitted the `Secure` attribute:

```go
    http.SetCookie(w, &http.Cookie{
        Name:     name,
        Value:    base64.RawURLEncoding.EncodeToString(raw),
        Path:     "/",
        MaxAge:   int(maxAge.Seconds()),
        SameSite: http.SameSiteLaxMode,
    })
```

## Remediation

True positive. Unlike the `HttpOnly` half of the same call (finding [2](./2.md), a false positive — the SPA must read
these cookies), the missing `Secure` attribute is a genuine gap: without it the browser also attaches these cookies to
plaintext `http://` requests to the same host. They carry no secrets, but there is no reason for them to travel over
cleartext, and the session cookie writer
([`writeChunked`](https://github.com/bitwise-media-group/patchy/blob/b377012bb2c0038ad49f1ca7c66ac4dce7fba9dc/internal/web/auth/cookies.go#L72-L80))
already sets `Secure` from the deployment's
[`cfg.Insecure`](https://github.com/bitwise-media-group/patchy/blob/b377012bb2c0038ad49f1ca7c66ac4dce7fba9dc/internal/web/auth/config.go#L61-L63)
flag.

The fix adds a `secure bool` parameter to `setJSONCookie`, threaded from that same flag, so the cookies are `Secure`
everywhere except the explicitly insecure plain-HTTP local-dev posture. `TestJSONCookieRoundTrip` now asserts the
attribute (and continues to assert the cookies stay SPA-readable, i.e. not `HttpOnly`).
